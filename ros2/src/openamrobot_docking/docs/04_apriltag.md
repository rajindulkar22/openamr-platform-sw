# AprilTag — Detection and Dock Placement

Everything about AprilTag setup: the detection pipeline (both real and simulated), tag configuration files, how to place the tag in the map frame, and tuning for accuracy.

This document consolidates what used to be `02_apriltag_ros.md` (detector config) and `05_dock_setup_map.md` (placing the tag in the map).

---

## 1. Detection pipeline

This package uses `apriltag_ros` to detect 36h11 tags. Two launch flavours coexist:

- **Real robot** — `launch/apriltag.launch.yml` runs `camera_ros` + `image_proc::RectifyNode` + `apriltag_ros::AprilTagNode` in a single component container.
- **Simulation** — `launch/apriltag_sim.launch.yml` runs only `apriltag_ros::AprilTagNode`, because the camera image comes from Gazebo via `ros_gz_bridge` and is already pinhole / undistorted.

### Real-robot pipeline

```
[ camera_ros::CameraNode ]
   ↓ /rgb_camera/image_raw, /rgb_camera/camera_info
[ image_proc::RectifyNode ]   (rectifies using calibration in /rgb_camera/camera_info)
   ↓ /image_rect
[ apriltag_ros::AprilTagNode ]
```

Composable nodes share a single process (component_container) for zero-copy intra-process comms.

### Simulation pipeline

```
[ Gazebo camera plugin ]
   ↓ via ros_gz_bridge → /camera/image_raw, /camera/camera_info
[ apriltag_ros::AprilTagNode ]
   (no rectify — sim camera has no distortion model)
```

Standalone node (not in a container).

### Why a separate sim launch

The real-robot launch loads `camera_ros::CameraNode` which expects a USB UVC camera and reads `~/.ros/camera_info/<name>.yaml` for calibration. In simulation neither exists — the image is generated by Gazebo and the intrinsics come from `<horizontal_fov>` in the SDF. Trying to use the real-robot launch in sim would either fail to start the camera node or load the wrong intrinsics.

## 2. Tag configuration files

| Hardware | File | Notes |
|---|---|---|
| Real robot | `config/tags_36h11.yaml` | `size: 0.0555` — **measure your printed tag** |
| Simulation | `simulation/config/tags_36h11_sim.yaml` | `size: 0.40` — matches the panel in `apriltag_dock/model.sdf` |

### Common parameters

```yaml
/**:
    ros__parameters:
        family: 36h11
        size: <meters>
        max_hamming: 0
        detector:
            threads: 1
            decimate: <1.0 sim, 2.0 real>
            blur: 0.0
            refine: True
            sharpening: 0.25
            debug: False
        pose_estimation_method: "pnp"
        tag:
            ids: [0]
            frames: [charging_dock_apriltag]
```

**`tag.frames`** is the TF frame the detector publishes for each detected tag. `detected_dock_pose_publisher` looks up the transform `parent_frame → tag.frames[0]` to publish `/detected_dock_pose`. This frame name must match `child_frame` in `docking_pose_publisher.yaml`.

## 3. Topic remaps

### Real robot

`launch/apriltag.launch.yml` performs these remaps:

```
RectifyNode:
  image       → /rgb_camera/image_raw
  camera_info → /rgb_camera/camera_info
AprilTagNode:
  /apriltag/image_rect    → /camera/image_rect    (output of Rectify)
  /apriltag/camera_info   → /rgb_camera/camera_info
```

If your camera publishes on different topics, update these remaps.

### Simulation

`launch/apriltag_sim.launch.yml` performs these remaps:

```
AprilTagNode:
  /apriltag/image_rect    → /camera/image_raw     (raw image, no rectification)
  /apriltag/camera_info   → /camera/camera_info
```

The sim camera is a perfect pinhole (no distortion), so we feed `image_raw` directly to apriltag's `image_rect` input.

---

## 4. Outputs

`AprilTagNode` publishes:

- **TF**: `<camera_optical_frame> → <tag.frames[i]>` for each detected tag.
- **`/apriltag/detections`** (`apriltag_msgs/AprilTagDetectionArray`): tag ID, center, corners, decision margin.

The TF is what downstream nodes consume. `detected_dock_pose_publisher` reads this TF and republishes the pose in the `map` frame as `/detected_dock_pose`.

## 5. Placing the AprilTag in the map frame

The docking pipeline needs to know where the tag is in the `map` frame. There are two ways to provide this:

1. **Static dock pose** — measure once, hardcode in config.
2. **Live detection pose** — `detected_dock_pose_publisher` computes it from TF every 100 ms.

Both pipelines use **both**: the static pose is the Nav2 goal target for the initial navigation, and the live detection drives the final approach precision.

### Where the static dock pose lives

Two files must agree on the dock pose.

#### `simulation/config/nav2_sim_full.yaml` (used by `opennav_docking`)

```yaml
docking_server:
  ros__parameters:
    docks: [home_dock]
    home_dock:
      frame: map
      pose: [4.0, 8.9, 1.5707]    # x, y, yaw in map
      type: home_dock_plugin
```

#### `config/dock_trigger.yaml` (used by `dock_trigger.py`)

```yaml
dock_pose_x: 4.0
dock_pose_y: 8.9
dock_pose_yaw: 1.5707
```

The third value is the **approach yaw**: the heading the robot has when docked (for our world setup, π/2 = facing north into the wall). Not to be confused with the AprilTag panel's SDF orientation, which is `-π/2` (the panel rotated 90° CW so it faces south).

## 6. How to measure the dock pose

### Real robot — from the robot itself

1. Run localization so `map → odom → base_link` exists.
2. Drive the robot to the dock and align it physically with the tag.
3. Read the tag pose in the `map` frame:

   ```bash
   ros2 run tf2_ros tf2_echo map charging_dock_apriltag
   ```

4. Note the translation `(x, y, z)` and rotation (quaternion).
5. Convert the quaternion to yaw — for a 2D-rotated tag, `yaw = 2 * atan2(qz, qw)`. Or use `tf_transformations.euler_from_quaternion()` in Python.
6. Update both YAML files (`openamrobot_docking.yaml` for the real pipeline, `dock_trigger.yaml` for the 4-phase sequencer).

### Real robot — from a SLAM map or floor plan

If you have a pre-built SLAM map:

1. Open the `.pgm` map in an image viewer.
2. Click on the tag location — the pixel coordinates × `resolution` + `origin` give you the map-frame xy.
3. Estimate yaw from the tag's normal direction (perpendicular to the wall it's mounted on, pointing into the room).

### Simulation — derived from the SDF world

The world file `simulation/worlds/docking_world.sdf` declares the tag at:

```xml
<include>
  <uri>model://apriltag_dock</uri>
  <name>apriltag_dock</name>
  <pose>0 4.9 0.3 0 0 -1.5707</pose>
</include>
```

That's `(x=0, y=4.9, z=0.3)` in the **world** frame, with model yaw `-π/2` so the panel faces south.

The robot spawns at world `(-4, -4)` yaw=0, so SLAM sets the map origin there with no rotation. Therefore:

- Tag map x = `0 + 4 = 4.0`
- Tag map y = `4.9 + 4 = 8.9`
- Approach yaw = `+π/2 = 1.5707` (robot must face north to enter the dock; this is the *opposite* of the panel's SDF model yaw of `-π/2` because the panel faces the robot, not the same direction as the robot's heading)

These are exactly the values in our YAML files.

## 7. Live detection vs static pose

### What `detected_dock_pose_publisher` does

```cpp
// src/detected_dock_pose_publisher.cpp (simplified)
Timer at publish_rate Hz:
    pose = tf_buffer.lookupTransform(parent_frame, child_frame, latest)
    publish PoseStamped(pose) on output_topic
```

Parameters in `config/docking_pose_publisher.yaml`:

```yaml
parent_frame: map
child_frame: charging_dock_apriltag
output_topic: detected_dock_pose
publish_rate: 10.0
```

This publishes `/detected_dock_pose` whenever the tag is visible (and the rest of the TF chain is up).

### How the 4-phase sequencer uses it

- **Phase 2** runs a closed-loop centring scan using TF `camera_rgb_optical_frame → charging_dock_apriltag` directly (bypassing the map-frame `/detected_dock_pose`), then collects `filter_num_samples` (40) fresh detections into a running-average tag pose with `filter_max_collect_time` (6.0 s) budget.
- **Phase 4** keeps reading `/detected_dock_pose` at every iteration during the advance. Each fresh reading folds into the running-average tag pose (true incremental mean — no arbitrary blend coefficient). The perpendicular yaw and lateral offset are recomputed from this average on every loop.

If `/detected_dock_pose` falls silent during the advance (tag occluded, lost), the running average is no longer updated but the controller keeps using the latest value. If the tag never produces a sample, phase 2 aborts cleanly with an error message.

### Ensuring static and live agree

If you measure with `tf2_echo map charging_dock_apriltag` and the result differs from your configured `dock_pose_*` by more than a few centimetres, investigate:

- Is your localization correctly aligned with reality? (For SLAM, the `map` origin depends on where the robot spawned.)
- Is the camera mounting correct? An offset between URDF `base_link → camera_link` and reality propagates into the live tag pose.
- Is `apriltag_ros`'s `size` correct? Wrong size = wrong distance estimate = wrong xy in map.

For the sim, all of these are determined by the SDF/URDF/YAMLs in this repo, so as long as you don't change the world layout, the values stay consistent.

## 8. Tag mounting tips

- **Vertical, perpendicular to the camera's expected approach direction.** A tilted tag works mathematically (the 4-phase sequencer uses the tag's `+Z` normal to derive the perpendicular yaw), but tilt amplifies pose noise.
- **Height matched to the camera's vertical FOV.** With `<horizontal_fov>1.2</horizontal_fov>` rad and 480 px height, the vertical FOV is ~0.95 rad (54°). At 1.5 m distance, the camera sees a vertical strip of ~1.5 m — a tag at z=0.3 (30 cm above ground) is visible whether the camera is at z=0.22 (sim) or higher.
- **Matte print surface.** Glossy print surfaces produce specular highlights that can defeat the detector.

## 9. Tuning for accuracy

Order of impact, biggest first:

1. **Tag size on the physical setup.** A 0.40 m tag at 1.5 m yields ~125 px of side at 640×480 → ~12 px/cell → sub-pixel solvePnP. A 0.0555 m tag at 1.5 m yields ~17 px/side → ~1.7 px/cell → very noisy.
2. **Camera resolution and FOV.** Smaller FOV (longer focal length) = more pixels per tag at a given distance. The sim's `horizontal_fov: 1.2 rad` (≈ 68°) is moderate.
3. **`detector.decimate`** — 1.0 = full resolution (most accurate), 2.0 = 2× downsampled (faster). Raise only if frame rate is a bottleneck.
4. **`detector.refine: True`** — sub-pixel corner refinement. Cheap, always helps.
5. **`pose_estimation_method`** — `pnp` (default) is more accurate than `homography` for non-planar rigs but slightly slower. Keep `pnp`.
6. **Lighting / contrast** — ensure the tag is well-lit with no glare; matte print surface beats glossy.

Beyond detection-side tuning, the 4-phase sequencer does **closed-loop image centring** in phase 2 (tag must be within 2° of image centre before sampling starts) and **incremental running-average filtering** throughout phase 2 and phase 4 (`filter_num_samples: 40`, then continuous updates) to further reduce noise — see [`08_sequencer_4phase.md`](08_sequencer_4phase.md).
